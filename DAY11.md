# インフラ＆セキュリティエンジニアリング学習記録

本日は、Linux（Ubuntu/WSL）環境のターミナルを活用し、インターネットの通信の仕組み（ネットワーク経路）と、データの最小単位である「16進数（バイナリデータ）」の解析について、実践を中心に深く学習しました。

## 🎯 本日の学習成果・習得コマンド

ネットワーク解析でよく使われる基本コマンドを学習しました。

| コマンド | 用途・学習内容 |
| :--- | :--- |
| `sudo apt install` | 不足しているネットワーク工具（`dnsutils`, `traceroute`）の自力インストール |
| `nslookup` | ドメイン（URL）から、DNSの仕組みを通じて本当の住所である「IPアドレス」を暴く |
| `ping -c 4` | 対象サーバーへ通信を行い、応答速度(ms)や疎通確認を行う |
| `traceroute -I` | パケットが目的地に届くまでの全ての交差点（ルーター）を追跡し、経路を丸裸にする |
| `od` / `echo -e` | 文字を16進数（Hex）の電気信号に変換（エンコード）し、それをまた元の文字に復元（デコード）する |

---

## 🔬 実践セッションの考察と気づき

### 1. ネットワーク経路の追跡（Google & GitHub）
-  データの往復時間（`time=〇〇 ms`）を解析したところ、GitHubへの通信が太平洋を越えることなく、わずか **7ms台** という国内並みの爆速で完了していることを突き止めました。
- 経路内の識別コードから、GitHubの親会社である **Microsoftの国内巨大データセンター（中継基地・キャッシュサーバー）** を通過していることが判明。現代の巨大IT企業が、世界規模のアクセスをどう分散・高速化しているのか、その裏側のインフラ構造を自分のコマンドで完全に可視化・証明することができました。

### 2. データ解析やログ調査で使われる16進数表現を学習した
- コンピューターの脳内言語である「16進数」に触れ、文字が裏側でどう電気信号に化けているかをパズル感覚で学びました。
- 悪意あるハッカーがセキュリティソフトの目を盗むために行う「コードの難読化」の心理を理解し、16進数の塊から元のメッセージを逆変換（デコード）する**セキュリティアナリストの基礎技術**を実践。
- `UTF-8` という文字コードの仕様（3バイトで1文字）を解剖し、コンピューターが数字の所属エリア（範囲）だけで「ひらがな」「カタカナ」「漢字」を厳格に区別しているスマートな仕組みを理解しました。
